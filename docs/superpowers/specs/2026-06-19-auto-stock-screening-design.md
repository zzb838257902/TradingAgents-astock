# TradingAgents-Astock 自动选股系统改造设计

> 日期：2026-06-19
> 状态：设计已确认，已完成独立审核并修订
> 适用项目：TradingAgents-Astock v0.2.14 及后续版本

## 1. 文档目的

当前系统要求用户输入一个股票代码，再由七类分析师、多空辩论、风险辩论和组合经理完成单股分析。该模式适合回答“这只股票怎么样”，但无法回答以下问题：

- 今天全 A 股中哪些股票值得进一步研究；
- 某个行业、概念或指数成分股中哪些标的更优；
- 不同持有周期下候选股是否不同；
- 候选股如何组成受约束、可解释的建议组合；
- 选股方法在历史上是否真正优于基准。

本设计在保留现有单股分析能力的前提下，增加完整的自动选股、板块选股、组合构建、回测和每日模拟验证流程。

本文是架构和开发设计，不承诺投资收益，也不包含券商自动下单。

## 2. 目标与非目标

### 2.1 建设目标

1. 默认覆盖全 A 股，排除 ST、*ST、退市整理、上市不足 60 个交易日、长期停牌及流动性过低股票。
2. 同时支持全市场、行业板块、概念板块、指数成分股和用户自定义代码列表。
3. 同时运行短线、波段和中长线三套独立策略。
4. 使用规则化多因子作为首版正式模型，预留机器学习排序接口。
5. 仅将定量模型筛出的少量候选股交给现有多 Agent 系统分析。
6. 输出 Top 10–20 候选及建议组合、权重上限、风险、止损条件和建议持有周期。
7. 支持盘后主筛、盘前事件复核、盘中候选监控三个运行阶段。
8. 建立 point-in-time 数据仓库和严格回测；只有通过 PIT 能力认证的数据集才能进入历史策略。
9. 默认适配个人单机环境，使用 DuckDB、SQLite 和文件缓存。
10. 数据源可插拔：首版提供免费默认方案并预留付费适配器；免费源不能满足的历史能力必须明确降级。

### 2.2 非目标

首版不包含：

- 券商实盘自动下单；
- 多用户权限、计费和账户系统；
- 分布式任务集群；
- 高频、逐笔或微秒级交易；
- 让 LLM 逐只扫描全部 A 股；
- 未经回测和人工审批的自动模型替换；
- 对收益率作确定性承诺。

## 3. 当前项目基线

### 3.1 已有能力

- `TradingAgentsGraph.propagate(ticker, trade_date)`：单股分析入口；
- 七类分析师：市场、情绪、新闻、基本面、政策、游资、解禁；
- 多空研究辩论、Research Manager、Trader、三方风险辩论和 Portfolio Manager；
- A 股行情、技术指标、财务、新闻、资金流、龙虎榜、解禁等工具；
- 多 LLM 供应商、结构化决策输出、检查点恢复和决策记忆；
- CLI 和 Streamlit Web UI。

### 3.2 与自动选股目标之间的缺口

- 没有全市场证券主表和历史股票池；
- 没有行业、概念、指数历史成分管理；
- 没有批量行情与财务数据仓库；
- 没有横截面因子计算和排名；
- 没有候选漏斗、组合优化和再平衡；
- 没有 point-in-time 回测；
- 部分现有接口虽然接收历史日期，实际返回当前数据；
- 现有交易约束主要存在于提示词，缺少确定性执行校验。

因此，自动选股不能通过循环调用现有 `propagate()` 实现。全市场约五千只股票若逐只运行 30–50 次 LLM 调用，将在成本、耗时、接口限流和可复现性上不可接受。

## 4. 方案比较与结论

### 4.1 方案 A：全市场规则评分

优点：便宜、快速、可解释、易回测。
缺点：对政策、公告、新闻和治理等非结构化信息利用不足。

### 4.2 方案 B：全市场机器学习排序

优点：可学习非线性关系，潜在预测能力较强。
缺点：高度依赖历史数据质量，容易发生前视偏差、样本泄漏和过拟合，首版风险较大。

### 4.3 方案 C：规则主筛 + 可插拔 ML + LLM 深度复核

这是本设计采用的方案。

```text
全市场或指定板块
  → 可交易性过滤
  → 三套规则化多因子评分
  → 可选机器学习排序
  → 综合候选 Top 30
  → 多 Agent 深度分析 Top 10–20
  → 风险约束和建议组合
  → 回测与每日模拟验证
```

核心原则：

- LLM 不承担全市场搜索，只承担候选股的非结构化验证；
- 三种时间周期保留独立分数，不压缩成无法解释的单一黑箱；
- 每层通过稳定数据契约通信，可以替换内部实现；
- 历史回测和实时运行使用相同策略接口，不复制两套逻辑。

### 5.2 模块边界

- Provider 只负责取得并转换数据，不计算策略分数；
- Repository 只负责持久化和按时间点查询；
- Factor 只接收标准化数据并输出因子值及质量信息；
- Strategy 组合因子，不直接访问网络；
- Ranker 只对候选排序，可在规则和 ML 实现之间切换；
- Agent Review 只处理 Top 候选，不改变原始因子；
- Portfolio 只依据候选评分和风险约束分配建议权重；
- Backtest 复用 Strategy 和 Portfolio，不调用实时 Provider。

这些边界使数据源、因子、模型、LLM 和风控可以独立优化。

### 5.3 与现有 `dataflows` 的迁移关系

新增数据层不能与 `tradingagents/dataflows/` 长期形成两套权威数据系统。唯一目标边界为：

```text
Agent / Screener / Backtest
          ↓
MarketDataRepository
          ↓
Provider adapters
```

迁移期间，现有 `dataflows` 工具作为 Repository 的兼容适配层；新功能不得直接依赖旧网站返回字符串。每迁移一个数据集，都要让单股分析和自动选股读取同一个标准化对象。迁移完成后，旧接口只保留 LangChain 工具包装职责。

## 6. 统一数据契约

### 6.1 必备时间字段

所有可用于策略的数据至少包含：

```text
symbol            标准证券代码
trade_date        数据归属交易日
as_of             数据内容对应时点
available_at      市场参与者实际可获得该数据的时间
source            数据来源
dataset_version   数据集版本
ingested_at       本系统抓取时间
quality_status    ok / stale / partial / conflict / invalid
pit_level         pit_required / current_only / best_effort
```

`available_at` 是防止前视偏差的核心字段。例：2025 年年报报告期为 2025-12-31，实际于 2026-03-20 发布，历史任务只能从 2026-03-20 之后使用。

数据集按历史能力分级：

- `PIT_REQUIRED`：已验证发布时间、历史覆盖和版本，可进入正式回测；
- `CURRENT_ONLY`：只能用于实时筛选或盘前复核，历史模式禁止读取；
- `BEST_EFFORT`：历史完整性不足，只用于探索性报告，不进入正式绩效验收。

首版免费数据的严格回测范围收敛到可靠日线、复权、证券状态、已披露财务数据及可验证的历史指数或行业成员。若历史概念、资金流、盈利预期、新闻或事件没有可靠快照，则只能标记为 `CURRENT_ONLY` 或 `BEST_EFFORT`。付费适配器接入并通过契约测试后，才可提升 PIT 等级。

### 6.2 数据分层

```text
raw         原始响应，只追加、不静默覆盖
normalized  统一字段、代码、单位和时间
feature     symbol + trade_date 的因子快照
signal      三策略分数、排名、贡献和风险标签
portfolio   建议权重、现金、调仓及原因
evaluation  未来收益、回测指标、模型版本表现
```

### 6.3 Provider 接口

Provider 至少定义：

```python
class MarketDataProvider(Protocol):
    def list_securities(self, as_of: date) -> list[Security]: ...
    def get_daily_bars(self, symbols, start, end) -> list[DailyBar]: ...
    def get_financials(self, symbols, available_before) -> list[FinancialRecord]: ...
    def get_industry_members(self, taxonomy, as_of) -> list[Membership]: ...
    def get_concept_members(self, as_of) -> list[Membership]: ...
    def get_index_members(self, index_code, as_of) -> list[Membership]: ...
    def get_events(self, symbols, published_before) -> list[MarketEvent]: ...
```

Provider 失败必须返回结构化错误或抛出约定异常，不能把 `Error ...` 字符串伪装成正常数据。应用层根据错误类型决定重试、降级或终止。

统一结果类型：

```python
DataResult[T](
    data=T | None,
    status=OK | EMPTY | STALE | ERROR,
    source="...",
    as_of="...",
    available_at="...",
    pit_level="...",
    errors=[],
)
```

`ERROR`触发供应商降级，`EMPTY`表示经过验证的合法空结果。旧数据函数返回的错误文本必须在接入自动选股前改造成该契约。

### 6.4 数据能力矩阵

阶段 0 必须先产出数据能力矩阵，至少列出：数据集、免费源、付费源、历史起点、更新频率、是否含 `available_at`、PIT 等级、字段完整率、许可和限流。没有能力矩阵的数据集不得写入正式因子清单。

## 7. 股票池和板块选股

### 7.1 股票池类型

系统同时支持：

- 全 A 股；
- 行业板块；
- 概念板块；
- 指数成分股；
- 用户自定义证券代码列表；
- 以上条件的交集或并集。

统一请求模型：

```python
UniverseQuery(
    market="A",
    industries=GroupFilter(values=["半导体"], mode="any"),
    concepts=GroupFilter(values=["人工智能", "机器人"], mode="any"),
    indices=GroupFilter(values=["中证1000"], mode="all"),
    custom_symbols=[],
    groups_combine="and",
    as_of="2026-06-19",
)
```

字段内部支持 `any/all`，字段之间支持 `and/or`，从而表达“电子 AND（人工智能 OR 机器人）AND 中证1000”。板块使用来源稳定 ID，中文名称只作为显示和解析入口。

### 7.2 板块解析

名称解析顺序：

1. 精确匹配标准名称或代码；
2. 匹配别名，如“AI”映射到“人工智能”；
3. 模糊匹配并返回候选；
4. 存在多个合理结果时要求用户选择，不静默猜测。

行业分类必须记录分类体系和版本。概念板块需记录来源及置信度。指数成分必须记录纳入和调出日期。

### 7.3 历史成分要求

回测不得使用今天的板块或指数成分替代历史成分。成员关系至少包含：

```text
group_type
group_code
group_name
symbol
effective_from
effective_to
source
```

若免费数据源无法提供可靠历史概念成分，该数据集只能用于实时筛选，必须从历史回测中禁用并显示能力降级。

### 7.4 历史证券有效期和幸存者偏差

证券主数据使用有效期模型：

```text
symbol
valid_from
valid_to
list_date
delist_date
name
board
status
st_flag
available_at
source
```

历史股票池必须包含分析日期当时已上市、后来退市的公司，并恢复当时的简称、ST、停牌和上市状态。股票池条件为 `listed_at(date) AND not_delisted_at(date) AND metadata_available_at <= signal_time`。只从当前证券列表倒推历史属于验收失败。

### 7.5 默认硬过滤

默认剔除：

- ST、*ST、退市整理和明确重大退市风险；
- 上市不足 60 个交易日；
- 当前停牌或近 20 日停牌比例超过阈值；
- 近 20 日日均成交额低于配置阈值；
- 长期无成交或关键行情严重缺失；
- 分析时点不可正常交易的证券。

北交所和其他特殊板块通过配置决定是否纳入。每只被排除股票都保存机器可读原因。

## 8. 三阶段运行流程

### 8.1 盘后主筛

1. 更新证券和板块主数据；
2. 增量拉取行情、财务、估值、行业和事件；
3. 执行完整性、时效性和异常检查；
4. 生成当日 point-in-time 因子快照；
5. 构建并过滤股票池；
6. 运行三类策略；
7. 综合排序形成 Top 30；
8. 对 Top 10–20 运行多 Agent 深度复核；
9. 构建建议组合；
10. 保存结果和报告，供下一交易日使用。

### 8.2 盘前复核

仅增量更新公告、政策、隔夜新闻、停复牌和风险事件。重大利空、ST、退市风险或停牌可触发降级或否决。盘前不重新训练模型，也不重新抓取全量历史数据。

### 8.3 盘中监控

只监控已有候选股的价格、成交量、涨跌停、资金流和止损条件。盘中不扫描全市场，不运行完整七分析师链路。

## 9. 因子体系

### 9.1 因子预处理

每个横截面因子依次执行：

1. 数据质量检查；
2. 缺失值保留标记并按配置处理；
3. 截面去极值；
4. 必要时行业中性化；
5. 必要时市值中性化；
6. 方向统一并转换为 0–100 分；
7. 保存原始值、标准化值、贡献值和扣分原因。

中性化不是所有因子的默认动作。行业相对强度等本身表达行业暴露的因子不得再次行业中性化。

### 9.2 短线策略（1–10 个交易日）

候选因子：

- 5/10/20 日价格动量；
- 成交额和换手率放大；
- 突破、均线排列、MACD、RSI；
- 涨停强度、连板和开板情况；
- 主力资金流、龙虎榜和机构席位；
- 题材热度与板块扩散；
- 新闻、公告和政策催化；
- 高波动、缩量涨停、尾盘异动等风险扣分。

首版初始权重：

```text
价格与成交量  30%
资金与龙虎榜  25%
题材和行业    20%
事件催化      15%
风险质量      10%
```

### 9.3 波段策略（2–12 周）

候选因子：

- 20/60/120 日趋势；
- 相对沪深 300 和所属行业的超额收益；
- 盈利预期上调；
- 机构关注和北向资金趋势；
- 行业景气与板块强弱；
- 估值相对自身历史分位；
- 波动率、最大回撤和趋势稳定性；
- 解禁、减持、质押等风险。

首版初始权重：

```text
趋势与相对强度  30%
盈利预期        20%
行业景气        20%
估值            15%
资金与风险      15%
```

### 9.4 中长线策略（3–24 个月）

候选因子：

- ROE、ROIC、毛利率和净利率；
- 营收、利润和自由现金流增长；
- 经营现金流与利润匹配度；
- 负债率、利息覆盖和财务安全；
- PE、PB、FCF 收益率及历史估值分位；
- 盈利稳定性和业绩修正；
- 行业地位、竞争优势和政策风险；
- 大股东减持、质押、解禁和治理风险。

首版初始权重：

```text
盈利质量    30%
成长能力    25%
估值水平    20%
财务安全    15%
治理与风险  10%
```

这些权重是待验证的初始参数，而不是经验真理。正式启用前必须通过滚动回测和稳定性检验。

## 10. 市场状态与策略融合

首版正式基线使用冻结的固定融合权重，三套策略同时单独回测和报告。市场状态动态权重作为独立实验，不计入 MVP 完成标准；只有通过消融和样本外验证后才可启用。

固定基线初始权重：

```text
短线 30%，波段 40%，中长线 30%
```

后续市场状态模型只能使用当时可见的指数趋势、市场宽度、成交额和波动率。不能把“弱市增加中长线权重”等同于自动降低风险，现金比例和组合 Beta 必须单独控制。

三类策略先分别保留 Top 100，再合并去重。多周期同时靠前的股票获得共振加分，但加分有上限，不能覆盖严重风险。

## 11. 候选漏斗与多 Agent 复核

### 11.1 候选规模

```text
全 A 股约 5000 只
→ 可交易股票池约 3500–4500 只
→ 三策略各 Top 100
→ 合并约 150–250 只
→ 综合定量 Top 30
→ Top 20 轻量结构化复核，Top 5 完整多 Agent 深度分析
→ 建议组合约 8–15 只
```

上述数量均可配置。板块样本较小时按比例缩减，不能为了满足数量强行纳入低质量股票。

### 11.2 现有 Agent 的新职责

- 验证定量信号的经济逻辑；
- 分析政策、公告、新闻和治理风险；
- 检查数据异常与信号冲突；
- 产生确认、降级或否决意见；
- 给出建议持有周期和主要催化剂；
- 不负责从全市场发现股票。

### 11.3 结构化复核输出

```text
quant_score
llm_confidence
data_quality
catalysts
risk_flags
veto
veto_reason
recommended_horizon
final_rating
```

首版 LLM 不直接修改量化分数，只能输出 `PASS`、`REVIEW_REQUIRED`、基于明确证据代码的 `VETO` 和文字解释。若以后实验调分，必须将 `llm_adjustment` 与原始分数分开保存，并单独回测稳定性。

历史模式下，Agent 禁止直接调用在线工具，只能读取冻结的 `AnalysisContext`：

```text
snapshot_id
symbol
as_of
allowed_datasets
factor_evidence
news_and_event_evidence
source_citations
```

工具路由增加 `execution_mode = live | historical`。历史模式访问实时 Provider 时立即报错。冻结上下文完成前，正式回测默认不启用 LLM，也不把 LLM 消融列为首版验收项。

批量 Agent 任务必须配置每日调用预算、token 预算、并发上限和单股超时，并以“股票＋日期＋快照＋模型＋提示词版本”缓存。单股失败不得拖垮整批任务。

### 11.4 质量门控调整

现有质量门控从“只提示”升级为状态机：

```text
PASS       继续
DEGRADED   使用降级源或降低置信度
RETRY      可重试错误，按退避策略重试
REJECTED   数据不足或重大冲突，停止该股票分析
```

质量门只检查本次实际启用的分析师报告。硬检查失败时先尝试数据源降级，不能依靠 LLM 自评掩盖缺失数据。

## 12. 组合构建与风险约束

默认输出 8–15 只股票的建议组合。组合构建器使用评分加权，并施加：

- 单股初始权重不超过 10%；
- 高波动短线股单股不超过 5%；
- 单一行业合计不超过 25%；
- 同一概念或高相关股票合计受限；
- 数据质量低于阈值不得入选；
- 重大减持、退市、财务异常可以否决；
- 组合保留 10%–50% 现金，由市场状态决定；
- 权重和金额满足实际交易单位约束。

组合请求必须包含：

```text
portfolio_value, max_participation_rate, min_position_value,
max_position_value, turnover_budget, cash_buffer
```

离散化为实际股数后必须再次校验现金、单股、行业和概念权重，避免四舍五入破坏约束。

首版使用可解释的启发式约束构建，不引入复杂优化器。后续可在相同接口下增加风险平价、均值方差或其他优化方法。

系统同时输出全市场排名和指定板块内排名。对板块选股，若股票数量过少或高度相关，允许输出少于 8 只并提高现金比例。

## 13. 调仓规则

- 盘后生成下一交易日建议组合；
- 正常情况下每周集中调仓，以控制换手；
- 短线子组合可以每日调整；
- ST、退市风险、重大事件和基本面恶化可以触发特殊调仓；
- 止损同时参考价格、波动率和投资逻辑失效；
- 每次调仓保存旧权重、目标权重、原因、数据版本和模型版本。

程序只生成建议，不直接提交券商订单。

## 14. 回测设计

### 14.1 基本流程

```text
历史数据准备
→ point-in-time 校验
→ 每个交易日生成股票池和因子
→ 当日收盘后信号形成次日订单
→ 模拟成交和交易约束
→ 更新持仓、现金和净值
→ 保存归因和绩效
```

### 14.2 成交和会计模型

至少支持：

- 次日开盘成交和 VWAP 成交两种模式；
- 佣金、过户费、印花税和滑点；
- 交易单位；
- T+1 可卖数量；
- 停牌不可成交；
- 一字涨停禁止买入、一字跌停禁止卖出的确定性保守模式；
- 按成交额参与率限制可成交数量；
- 分红、送股、拆并股和复权；
- 退市和长期停牌异常退出。

交易费率和板块交易规则必须按生效日期配置，不能用一套当前规则覆盖全部历史时期。

### 14.3 比较基准

- 与实际股票池风格匹配的沪深300、中证500、中证1000、中证全指或行业指数；
- 对应股票池等权组合；
- 简单动量策略；
- 三套独立策略；
- 多策略融合；
- 不经过 LLM 与经过 LLM 复核的组合。

### 14.4 指标

- 累计和年化收益；
- 年化波动率；
- 最大回撤；
- Sharpe、Sortino、Calmar；
- 超额收益和信息比率；
- 月度胜率、选股胜率和盈亏比；
- 换手率和交易成本占比；
- 行业、风格和市值暴露；
- Top 10/20/50 分层收益；
- 因子 IC、Rank IC 及稳定性；
- 不同市场状态下的表现。

### 14.5 防过拟合

- 使用按时间顺序划分的训练、验证和完全留出测试集；
- 使用 walk-forward 滚动验证；
- 参数搜索只访问训练和验证期；
- 测试期只在最终候选模型上运行；
- 检查参数扰动、时间段、行业和股票贡献集中度；
- 每个模型和配置保存版本，不覆盖历史结果。

### 14.6 首版验收标准

首次正式回测前冻结以下门槛；若后续修改，必须创建新验收版本并保留旧结果：

- 完全留出测试期不少于 3 年，并覆盖至少一个下跌超过 20% 的宽基指数阶段；
- 扣除全部交易成本后，年化超额收益大于 2 个百分点；
- 至少 60% 的完整自然年度取得正超额收益；
- 策略最大回撤不得比基准多 5 个百分点以上；
- Top 组合相对 Bottom 组合的年化收益差大于 3 个百分点；
- 月度 Rank IC 均值大于 0.02，且正 IC 月份比例不低于 55%；
- 任一行业贡献不得超过总超额收益的 40%，前五只股票贡献不得超过 50%；
- 关键权重统一上下扰动 10% 后，测试期扣费超额收益仍为正；
- LLM 复核组合相对无 LLM 组合至少改善一个主要指标，且不得让年化超额或最大回撤中的另一个指标恶化超过 2 个百分点，否则 LLM 不进入正式组合。

这些是首版工程准入线，不代表统计意义上的收益保证。正式研究报告还应同时给出置信区间、分期结果和多重检验说明。

验收还必须报告容量、成本敏感性及 block bootstrap 置信区间。若策略股票池不是大盘股，不能只用沪深300作为唯一基准。LLM 增益还必须覆盖其调用成本，并在多个时间窗稳定。

## 18. 调度和幂等性

首版使用 APScheduler 或操作系统定时任务，不引入 Redis 和分布式队列。

每个任务由以下字段唯一标识：

```text
job_type + trade_date + universe_hash + config_version + model_version
```

重复运行默认复用已完成结果；使用 `--force` 才允许重新计算。部分失败可从股票或阶段级检查点恢复。

东财等接口继续遵守全局限流。批量采集必须优先使用批量端点、缓存和增量更新，禁止为每只股票重复抓取相同市场数据。

## 19. 开发阶段

独立审核后改为先完成最小纵向闭环，再扩展数据广度。

### 阶段 0：数据可得性审计

交付：免费/付费数据能力矩阵、PIT 分级、历史覆盖和许可说明。
完成标准：正式因子所需数据均有明确来源和能力等级。

### 阶段 1：统一数据契约和证券有效期

交付：`DataResult`、时间语义、交易日历、证券有效期主表、Repository 边界及旧 `dataflows` 迁移规则。
完成标准：历史日期可包含后来退市股票，历史模式拒绝实时数据。

### 阶段 2：最小历史数据仓库

交付：日线、复权、停牌、可靠财务披露数据及 DuckDB 迁移。
完成标准：所有数据通过 PIT 契约测试。

### 阶段 3：最小规则策略和确定性回测（MVP）

交付：全市场/行业/指数股票池、一个动量策略、一个财务质量策略、固定融合基线、确定性成交模型。
完成标准：形成从历史数据到组合净值的可验证纵向闭环。

### 阶段 4：三策略和组合约束（V1）

交付：完整短线、波段、中长线规则因子，板块内外双重排名和容量约束。
完成标准：三策略独立回测并通过冻结验收协议。

### 阶段 5：概念和事件实时筛选（V1.1）

交付：概念板块、新闻、公告、资金流等实时或 best-effort 数据。
完成标准：能力降级明确，不污染历史正式回测。

### 阶段 6：盘后自动任务和报告（V1.2）

交付：盘后任务、幂等恢复、CSV/Markdown/Web 报告。
完成标准：个人单机可稳定完成每日盘后选股。

### 阶段 7：冻结上下文后的 LLM 风险复核（V2）

交付：`AnalysisContext`、Top 20 轻量复核、Top 5 深度分析、预算和缓存。
完成标准：历史模式无实时污染，消融实验可重复。

### 阶段 8：后续扩展（V3）

交付：ML 排序、盘前复核、盘中观察提醒和可验证市场状态模型。
完成标准：各能力独立通过样本外或运行稳定性验收。

盘中首版只承担候选观察和提醒，不承担实时交易决策。

## 20. 测试策略

### 20.1 单元测试

- 字段转换和单位归一；
- 股票池交并集和别名解析；
- 硬过滤原因；
- 因子公式和方向；
- 去极值、中性化和缺失处理；
- 三策略权重；
- 组合约束和交易单位。

### 20.2 数据契约测试

每个 Provider 使用固定样本响应验证标准化结果。接口字段变化时必须失败并报警，不能静默生成全空数据。

### 20.3 时间点测试

- 财报在发布日期前不可见；
- 新闻在发布时间前不可见；
- 历史板块只包含当时成员；
- 历史任务拒绝实时快照；
- 因子窗口不会读取目标日之后数据。

### 20.4 集成和回归测试

- 板块输入到候选组合的完整链路；
- 数据源超时、限流、空数据和降级；
- 批量 Agent 中断恢复；
- 保留现有单股分析、CLI 和 Web 功能；
- 配置版本和数据库迁移。

### 20.5 回测正确性测试

- T+1 可卖数量；
- 涨跌停和停牌；
- 手数和现金不足；
- 交易费用；
- 分红、复权和公司行动；
- 不同成交模式；
- 退市和长期停牌。

### 20.6 性能测试

记录全市场数据更新、因子计算、候选生成、Agent 分析和报告生成的耗时、内存及接口调用量。性能目标在阶段 2 完成数据规模基准后确定。

## 21. 可观测性和故障处理

每次运行保存：

- 任务状态和阶段耗时；
- 数据源调用量、失败率和限流次数；
- 数据缺失、过期和冲突数量；
- 每层候选数量；
- Agent 调用次数、token 和成本；
- 使用的数据、配置和模型版本；
- 最终结果和失败原因。

错误处理分类：

- 瞬时网络错误：指数退避重试；
- 数据源限流：遵守 Retry-After 或内部节流；
- 字段结构变化：契约失败并停止相关数据集更新；
- 部分股票缺失：标记降级，不影响其他股票；
- 关键市场数据缺失：停止当日正式选股，避免输出伪精确结果；
- LLM 不可用：仍可输出定量候选，但明确标记未经过深度复核。

## 22. 安全和合规

- 所有输出保留“研究用途，不构成投资建议”；
- API Key 仅从环境变量读取；
- 用户输入的股票代码、板块名和文件路径必须校验；
- 原始新闻和报告遵守来源使用条款，不大规模再分发全文；
- 日志中不得输出 API Key 或完整认证头；
- 自动选股与实盘交易保持明确边界。

## 23. 后续优化接口

系统允许在不改变上层调用方式的情况下优化：

- 免费数据源替换为付费数据源；
- 新增或删除因子；
- 调整或动态学习策略权重；
- 规则 Ranker 替换为 LightGBM Ranker；
- 市场状态规则替换为统计模型；
- 更换 LLM、Agent 角色或提示词；
- 启发式组合替换为其他优化器；
- APScheduler 替换为服务器任务系统；
- 增加更多行业分类、概念来源和指数体系。

所有替换必须通过相同的数据契约、回测和版本化验收流程。

## 24. 完成定义

本改造只有同时满足以下条件才视为首版完成：

1. 原有单股分析功能保持可用；
2. 可从全 A 股和指定行业、概念、指数中生成股票池；
3. 可执行三套独立策略和综合排名；
4. 可对 Top 候选执行多 Agent 深度复核；
5. 可输出受约束、可解释的建议组合；
6. 盘后、盘前和盘中任务可独立运行并恢复；
7. 历史回测通过时间点和交易规则正确性测试；
8. 输出包含数据、配置和模型版本，可重复生成；
9. 数据源失败时明确降级或停止，不输出静默错误；
10. 自动选股结果通过事先定义的样本外验收标准。

## 25. 主要风险与控制措施

| 风险 | 影响 | 控制措施 |
|---|---|---|
| 免费接口不稳定 | 数据缺失或任务中断 | Provider 契约、缓存、限流、重试、降级、付费适配器 |
| 历史数据不具备 point-in-time 属性 | 回测虚高 | `available_at`、历史成分、实时数据隔离、时间点测试 |
| 因子过多和权重调参 | 过拟合 | 少量首版因子、walk-forward、留出测试、参数稳定性检查 |
| LLM 幻觉或偏见 | 错误调分和风险判断 | Top 候选限定、结构化证据、调分上限、可关闭和消融实验 |
| 板块高度集中 | 组合伪分散 | 行业、概念和相关性约束，允许提高现金比例 |
| 全市场批量任务过慢 | 无法在盘后窗口完成 | 批量接口、增量更新、向量化因子、缓存，LLM 仅分析少量候选 |
| 规则随监管变化 | 历史或当前成交模拟错误 | 按生效日期版本化交易规则 |
| 单机数据库损坏或磁盘不足 | 历史结果丢失 | 原子写入、备份、空间监控、数据库迁移和校验 |

## 26. 独立审核记录

独立子智能体对本设计进行了批判性审核，结论为“有条件通过”。本次修订吸收了以下主要意见：

1. 将数据分为 `PIT_REQUIRED`、`CURRENT_ONLY` 和 `BEST_EFFORT`，不再承诺免费源支持全部历史能力；
2. 增加证券有效期模型，避免遗漏后来退市股票造成幸存者偏差；
3. 增加冻结 `AnalysisContext` 和 historical 模式防火墙，阻止 Agent 引入实时数据；
4. 明确新 Repository 是唯一目标数据边界，旧 `dataflows` 逐步迁移为兼容层；
5. 将动态市场权重后移，首版使用固定基线；
6. 取消 LLM 直接调分，首版仅允许确认、人工复核和证据化否决；
7. 将完整多 Agent 分析缩减到 Top 5，并增加预算、缓存和失败隔离；
8. 扩展板块查询逻辑，保留全市场绝对分和板块内相对排名；
9. 增加账户规模、成交容量和确定性涨跌停规则；
10. 将原七阶段大版本拆为阶段 0–8 的 MVP/V1/V2/V3 纵向里程碑。

审核同时指出，历史概念成分、历史资金流、历史盈利预期和完整新闻事件很可能需要付费或自行长期积累的数据。相关能力在数据矩阵确认前不得进入正式回测完成标准。

本设计的下一步是在用户确认书面规格后，编制逐任务、逐文件、带测试先行步骤的实施计划。实施计划不应一次性铺开七个阶段，而应按阶段建立可验证的里程碑。
